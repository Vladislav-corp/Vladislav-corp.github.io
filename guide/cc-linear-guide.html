Управляй Linear прямо из Claude Code

Коннектор по протоколу MCP даёт агенту прямой доступ к задачам, а хуки превращают его из «помощника» в надёжный конвейер: форматирование, тесты, гварды и обновление статусов — автоматически.

⚡ Подключение ~60 сек 🔑 Без API-ключа 🪝 17 типов событий хуков 🤖 Авто-цикл тикет → код → ревью

Подключение коннектора

MCP (Model Context Protocol) — открытый протокол, через который агент читает рабочее пространство Linear напрямую: заголовок, описание, приоритет, метки, исполнителя, контекст проекта. Подключить можно тремя путями: через приложение Claude, прямо в терминале или командой CLI.

Способ 1 · приложение

Claude → Customize

  1. Откройте Customize в приложении Claude
  2. Раздел Connectors
  3. В списке выберите Linear
  4. Нажмите подключение — вход через браузер
Способ 2 · терминал

Claude Code → /plugins

  1. В сессии введите /plugins
  2. Вкладка Discover
  3. Найдите и выберите Linear
  4. Enter → Enter to auth для входа

Способ 3 · командой CLI

Для тех, кто предпочитает консоль без интерактивного меню — регистрация удалённого MCP-сервера одной командой. Настройка одинакова и для CLI, и для IDE-расширения.

Терминал · регистрация сервера
# добавляем удалённый MCP-сервер Linear
claude mcp add --transport http linear-server https://mcp.linear.app/mcp
Внутри сессии Claude Code · OAuth
# откроется браузер для входа в Linear — ключ не нужен
/mcp
🔍

Проверка

Команда /mcp также показывает список подключённых серверов и их статус. Если Linear в списке зелёный — всё готово.

⚠️

Remote-MCP пока ранний

Соединение иногда срабатывает не с первой попытки — повторите авторизацию. Токены истекают: при ошибках доступа снова запустите /mcp.

Создание задач

Когда коннектор подключён, задачи создаются и обновляются обычным текстом — никаких форм и ID проектов наизусть. Агент сам подставит нужные поля.

Создай баг в проекте Payments: «Падение при возврате средств на крайних значениях». Приоритет High, метка backend, назначь на @alex.
Заведи задачу «Добавить тёмную тему в настройки», опиши acceptance-критерии и привяжи к текущему циклу.

Что можно задавать в одном запросе

  • Заголовок и описание (с markdown и acceptance-критериями)
  • Приоритет: Urgent / High / Medium / Low
  • Метки, исполнитель, проект, цикл, срок
  • Связанные задачи (blocks / blocked by / related)
💡

Лайфхак: пакетное создание

«Разбей эту фичу на 5 подзадач по слоям (API, БД, UI, тесты, доки) и создай их все в проекте X, связав как подзадачи эпика ENG-1200».

Хуки: основы

Хуки — это shell-команды или скрипты, которые срабатывают в определённые моменты жизненного цикла сессии. Главное: они не зависят от того, «вспомнит» ли модель что-то сделать — выполняются всегда при совпадении условия. Это детерминированный слой поверх агента.

Три компонента любого хука

  • Событие — когда срабатывает (PreToolUse, PostToolUse, Stop, …)
  • Matcher — regex-фильтр: например, только инструмент Bash или Edit|Write
  • Action — что выполнить: shell-команда или промпт к лёгкой модели

Ключевые события

СобытиеКогдаМожет блокировать?
SessionStartНачало сессии—
UserPromptSubmitОтправка запросада
PreToolUseПеред вызовом инструментада (exit 2)
PostToolUseПосле инструментанет, но даёт фидбек

Где настраивать

Интерактивно — командой /hooks, либо вручную в JSON. Глобально: ~/.claude/settings.json. Для команды (в репозитории): .claude/settings.json — так гварды разделяет вся команда.

🔒

Безопасность

Хуки запускаются с правами вашего пользователя, без песочницы. Держите их короткими, явными и проверяйте ввод. Код выхода: 2 — блок, 0 — разрешить, 1 — только предупреждение.

Рецепты хуков

Готовые к вставке в .claude/settings.json. Каждый самодостаточен.

1 · Авто-форматирование после правок

PostToolUse · самый популярный
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit|MultiEdit",
      "hooks": [{
        "type": "command",
        "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""
      }]
    }]
  }
}

2 · Блок опасных команд

PreToolUse · гард на Bash (exit 2 = блок)
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "grep -qE 'rm -rf|:(){' <<< \"$CLAUDE_TOOL_INPUT\" && exit 2 || exit 0"
      }]
    }]
  }
}

3 · Гейт на тесты перед завершением

Stop · не дать закончить с красными тестами
{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "npm test --silent || { echo 'Тесты падают — почини до завершения'; exit 2; }"
      }]
    }]
  }
}
♻️

Бесконечный Stop-цикл

Чтобы хук не зацикливался, проверяйте поле stop_hook_active во входном JSON: если true — сразу exit 0.

4 · Уведомление в Slack/на десктоп

Notification · алерт, когда нужен ввод
{
  "hooks": {
    "Notification": [{
      "hooks": [{
        "type": "command",
        "command": "curl -s -X POST $SLACK_WEBHOOK -d '{\"text\":\"Claude ждёт ответа\"}'"
      }]
    }]
  }
}

Авто-цикл Linear ↔ код

Самое интересное — соединить MCP и хуки. Хуки умеют матчиться и на MCP-инструменты, так что можно ставить гварды на вызовы Linear и автоматизировать обновление статусов.

1 · ЧИТАЕТ
тикет ENG-1234 из Linear (MCP)
→
2 · ПРАВИТ
код; PostToolUse авто-форматирует
→
3 · ГЕЙТ
Stop-хук гонит тесты
→
4 · ОБНОВЛЯЕТ
статус → In Review + коммент
Возьми ENG-1234 из Linear, реализуй по существующим паттернам кода, прогони тесты и переведи задачу в In Review с комментарием о внесённых изменениях.

Гард на запись в Linear

PreToolUse · подтверждение перед изменением задач
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "mcp__linear.*",
      "hooks": [{
        "type": "command",
        "command": "./scripts/confirm-linear-write.sh"
      }]
    }]
  }
}
🧩

SessionStart для контекста

Повесьте на SessionStart скрипт, который подтягивает в контекст текущий спринт из Linear — агент сразу знает приоритеты, не спрашивая.

Как я это использую в работе

Реальный процесс, который закрыл для меня главную боль — потерю контекста по проектам. Я больше ничего не держу в заметках, файлах или «памяти»: единственный источник правды — Linear, а Claude Code сам его читает и пополняет.

🪝

Ключ ко всему — хук в конце сессии

На завершении работы у меня стоит хук: Claude обязательно идёт в Linear и сверяет статус проекта. Поэтому каждая сессия заканчивается актуальной картиной — без моего участия.

Весь мой цикл — одной фразой

Зайди в проект, возьми задачу и реализуй её.

Дальше он делает всё сам — потому что у него есть прямой доступ к Linear через коннектор и он понимает, в каком проекте мы работаем:

1 · УЗНАЁТ ПРОЕКТ
понимает, что это за проект, и видит его статус
→
2 · ЧИТАЕТ КОНТЕКСТ
description, актуальные задачи и проблемы
→
3 · БЕРЁТ ЗАДАЧУ
ту, что сейчас In Progress, и работает в её рамках
→
4 · ЗАКРЫВАЕТ
переводит в Done, описывает сделанное, шлёт в проект

Почему это так классно работает

  • Контекст проекта сам накапливается: какие работы проведены, текущий статус, актуальные проблемы.
  • Идеи и фичи я записываю прямо задачами в Linear — а потом беру их в работу.
  • Не нужно держать знания в «памяти» Claude, в файлах или заметках — источник один.
  • Контекст всегда актуален: он обновляется в момент работы, а не задним числом.
✅

Что это закрывает

Большую проблему контекста в рабочих проектах. Знания не теряются между сессиями: связка «задача → описание → результат» сохраняется, и любой следующий заход начинается с полной картины — легко, просто и всегда свежо.

Хук, который запускает синк с Linear

Stop · сверка с проектом перед завершением
{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "echo '{\"hookSpecificOutput\":{\"additionalContext\":\"Перед завершением: открой текущий проект в Linear, сверь статус, переведи задачу в Done и опиши, что сделано.\"}}'"
      }]
    }]
  }
}
♻️

Чтобы не зациклиться

Stop-хук может срабатывать повторно — проверяйте stop_hook_active во входном JSON и при true сразу выходите с exit 0.

Лайфхаки

  • Пусть Claude сам напишет хук: «Напиши хук, который запускает eslint после каждой правки файла».
  • Командные гварды в репо. Коммитьте .claude/settings.json — у всех одинаковые правила форматирования и безопасности.
  • Один хук в неделю. Внедряйте по одному: авто-формат, затем запрет rm -rf, затем гейт на тесты.
  • Ссылайтесь на тикеты по ID. «Сделай как в ENG-1200, но для профиля» — агент подтянет контекст сам.
  • Связывайте задачи сразу. Просите проставлять blocks/related при создании — меньше ручной уборки.
  • claude --debug — если хук не срабатывает как ожидалось, смотрите, что и когда триггерится.
  • Разделяйте контекст. Linear MCP даёт контекст задач, инструмент поиска по коду — контекст кода; агент оркестрирует обоими.
